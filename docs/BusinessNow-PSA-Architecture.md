## 1. System Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                        USER'S BROWSER                           │
│                                                                 │
│   React + Vite SPA (port :5000)                                 │
│   ├── Wouter (client-side routing)                              │
│   ├── React Query (server-state cache + mutations)              │
│   ├── Zod (request/response validation)                         │
│   ├── Orval-generated hooks  ◄── OpenAPI codegen                │
│   └── Recharts (data visualisation)                             │
└───────────────────────┬─────────────────────────────────────────┘
                        │  HTTP / JSON  (BASE_URL prefix)
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│                  EXPRESS 5 API SERVER (port :8080)              │
│                                                                 │
│   ├── RBAC middleware  (requirePM / requireFinance / requireAdmin)│
│   ├── Route handlers  (/api/*)                                  │
│   ├── Zod request validation                                    │
│   ├── logAudit() helper  →  audit_log table                     │
│   └── Auto-triggers                                             │
│        ├── Opportunity probability ≥70% → soft allocation       │
│        ├── Milestone task completed  → draft invoice            │
│        └── Timesheet approve/reject  → notification             │
└───────────────────────┬─────────────────────────────────────────┘
                        │  Drizzle ORM queries
                        ▼
┌─────────────────────────────────────────────────────────────────┐
│               PostgreSQL Database                               │
│   37 tables across 11 domain modules                            │
└─────────────────────────────────────────────────────────────────┘
```

### Design Principles

| Principle | Implementation |
|-----------|---------------|
| Single source of truth | OpenAPI spec → Orval codegen → React Query hooks + Zod schemas |
| Contract-first API | `lib/api-spec` defines the full OpenAPI YAML |
| Type safety end-to-end | Drizzle infers DB types; Zod validates at the API boundary; TypeScript throughout |
| Soft delete | Projects use `deletedAt` timestamp instead of hard deletion; restore endpoint available |
| Audit trail | `logAudit()` called from all write paths for tasks, projects, opportunities, allocations, invoices, change orders |
| RBAC | Role header `x-user-role` checked by middleware on write routes |

```
workspace/
├── lib/
│   ├── db/                   @workspace/db
│   │   └── src/schema/       — Drizzle table definitions (37 tables)
│   ├── api-spec/             @workspace/api-spec
│   │   └── openapi.yaml      — OpenAPI 3.0 specification
│   ├── api-zod/              @workspace/api-zod
│   │   └── src/generated/    — Zod schemas auto-generated by Orval
│   └── api-client-react/     @workspace/api-client-react
│       └── src/generated/    — React Query hooks auto-generated by Orval
│
├── artifacts/
│   ├── businessnow/          @workspace/businessnow   (React + Vite SPA)
│   │   └── src/
│   │       ├── pages/        — One file per module (13 pages)
│   │       ├── components/   — Shared UI components
│   │       └── contexts/     — CurrentUserProvider (session state)
│   └── api-server/           @workspace/api-server    (Express 5)
│       └── src/routes/       — One file per domain (30 route files)
```

### Code Generation Pipeline

```
lib/api-spec/openapi.yaml
        │
        │  pnpm --filter @workspace/api-spec run codegen
        ▼
lib/api-zod/src/generated/api.ts        ← Zod schemas
lib/api-client-react/src/generated/     ← React Query hooks
```

## 3. API Layer & Data Flow

### Request → Response Lifecycle

```
Browser                API Server              Database
  │                        │                      │
  │── GET /api/projects ──►│                      │
  │                        │── Drizzle SELECT ───►│
  │                        │◄── rows ─────────────│
  │                        │── mapProject(row)     │
  │                        │── Zod.parse(mapped)   │
  │◄── JSON array ─────────│                      │
  │                        │                      │
  │── POST /api/tasks ─────│                      │
  │   (x-user-role: PM)    │── requirePM() check  │
  │                        │── Zod body validate  │
  │                        │── Drizzle INSERT ───►│
  │                        │── logAudit() ────────│
  │                        │── auto-trigger? ─────│
  │◄── 201 Created ────────│                      │
```

### RBAC Middleware Chain

```
requireAdmin  — Admin only
requirePM     — Admin or Project Manager
requireFinance — Admin, PM, or Finance
requireRM     — Admin, Resource Manager (for allocation approval)
```

### 4.1 Users & Access Control

**Purpose:** Stores all team members, their roles, department, billing rates, skills, and secondary roles for the role-switcher feature.

#### `users`

| Column | Type | Notes |
|--------|------|-------|
| `id` | serial PK | Auto-increment |

| `is_active` | integer | 1 = active, 0 = deactivated |
| `created_at` | timestamptz | |

**Key API endpoints:**  
`GET /api/users` · `POST /api/users` · `PATCH /api/users/:id` · `DELETE /api/users/:id`

## 5. Module Flow Diagrams

### 5.1 CRM Pipeline Flow

```
[Prospect Created]
       │
       │  Status: New → Qualified → Proposal → Negotiation
       │
       ▼
[Convert to Customer]
       │
       ├──► Creates Account record
       └──► Prospect.convertedAccountId = Account.id
                    │
                    ▼
           [Create Opportunity]
           linked to Account
                    │
          Stage: Discovery → Qualified → Proposal → Negotiation
                    │
                    │  Stage = Won
                    ▼
           [Create Project]
           Opportunity.projectId = Project.id
           Project.opportunityId = Opportunity.id
```

### 5.2 Project Lifecycle Flow

```
[Project Created]
(from template or scratch)
       │
       ├── Phases added
       │      └── Tasks added to phases
       │              ├── Sub-tasks (parentTaskId)
       │              ├── Dependencies (task_dependencies)
       │              └── Checklist items
       │
       ├── Team allocated (allocations)
       │      └── Soft → Hard when confirmed
       │
       ├── Time logged (time_entries)
       │      └── Via timesheets (weekly)
       │
       ├── Change orders (if scope changes)
       │
       └── Invoices generated
              ├── From approved timesheet
              ├── From payment milestone
              └── From billing schedule
```

### 5.3 Time Tracking & Approval Flow

```
User logs time entries
(project + task + category + billable flag)
       │
       ▼
Weekly Timesheet created (Draft)
       │
       │  User submits
       ▼
Timesheet status = Submitted
       │
       ├──► Notification sent to PM/approver
       │
       │  PM approves                    PM rejects
       ▼                                     ▼
status = Approved                    status = Rejected
time_entries.approved = true         rejection_note saved
Notification to user                 Notification to user
       │
       ▼
"Generate Invoice" available
→ POST /api/invoices/from-timesheet/:id
→ Draft Invoice created
```

### 5.4 Resource Request Flow

```
PM identifies staffing gap
       │
       ▼
Resource Request created
(role, skills, dates, hrs/week, priority)
       │
       ▼
Resource Manager reviews
       │
       ├── Reject → status = Rejected, reason stored
       │
       └── Approve → Assign user
                       │
                       ▼
                 status = Fulfilled
                 Allocation created automatically
                 (hard allocation, same dates/hrs)
```

### 5.5 Finance Flow

```
Project Budget set
       │
       ├── Change Orders (Pending → Approved)
       │      └── Approved COs increase effective budget

       ├── Rate Card applied (per-role billing rates)
       │
       ├── Time entries → Billable hours tracked
       │
       ├── Billing Schedules
       │      ├── Date-triggered → auto-create Draft Invoice
       │      └── Milestone-triggered → fires on task completion
       │

       │

       │
       └── Reports
```

## 6. Role-Based Access Control

### Roles

| Role | Description |
|------|-------------|
| **Admin** | Full access to all modules and settings |
| **PM** (Project Manager) | Create/edit projects, approve timesheets, manage tasks |
| **Resource Manager** | Approve/reject resource requests, manage allocations |
| **Finance** | Manage invoices, view financial reports |
| **Viewer** | Read-only access across modules |
| **Consultant / Engineer / etc.** | Delivery roles; can log time and view assigned projects |

### Secondary Roles

A user can have a primary role plus one or more `secondary_roles`. The sidebar role-switcher lets the user switch their active role within the same session (persisted to `localStorage`). Configured via Admin → Users → User Configuration tab.

### Middleware Matrix

| Route Type | Required Role |
|-----------|---------------|
| `POST/PATCH/DELETE /api/projects` | PM |
| `GET /api/projects/deleted` | Admin |
| `POST /api/projects/:id/restore` | Admin |
| `POST/PATCH/DELETE /api/accounts` | PM |
| `POST/PATCH/DELETE /api/invoices` | Finance |
| `DELETE /api/users/:id` | Admin |
| `PATCH /api/users/:id/secondary-roles` | Admin |
| `POST/PATCH /api/allocations` | Resource Manager |
| All GET endpoints | Any authenticated user |

## 7. Key Business Workflows

### Workflow A — Opportunity to Revenue

```
Prospect → Account → Opportunity → Won
→ Create Project (with template)
→ Allocate Team (hard allocations)
→ Phases & Tasks defined
→ Team logs time weekly
→ Timesheets submitted & approved
→ Invoices generated (milestone / timesheet / schedule)
→ Invoices approved & paid
→ Revenue entries recognised
→ Margin reported
```

### Workflow B — Automatic Soft Allocation on High-Probability Deal

```
Opportunity.probability updated to ≥ 70%
       │
       └── API auto-inserts allocation:
           userId = opportunity.ownerId
           projectId = opportunity.projectId
           isSoftAllocation = true
           logged to audit_log
```

### Workflow C — Payment Milestone Auto-Invoice

```
Task marked as Completed
  AND task.isMilestone = true
  AND task.milestoneType contains "Payment"
       │
       └── API auto-creates Draft Invoice:
           amount = project.budget
           description = "Milestone: {task.name}"
           logged to audit_log
```

### Workflow D — CSAT Collection

```
Project delivery milestone completed
→ CSAT form sent to client
→ Client submits star rating (1–5) + comment
→ csatResponses stored per project + task
→ CSAT tab on project shows:
   - Average rating card
   - Star distribution bar chart
   - Recent feedback feed
```
